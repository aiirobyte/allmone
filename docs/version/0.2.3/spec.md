# allmone v0.2.3 Spec

Last updated: 2026-05-11
Status: Complete

## Version Goal

Define API-key Provider output model names before the broader v0.3.0 model resource inventory work. Every API-key or OpenAI-compatible Provider model that allmone writes into CLIProxyAPI config should have a deterministic, Provider-specific alias:

```yaml
- name: gpt-5.5
  alias: mimo_a-gpt-5.5
  fork: true
```

The alias enables specific Provider calls through the local OpenAI-compatible output port. `fork: true` also keeps the raw upstream model ID available for CLIProxyAPI's built-in shared routing or load-balancing behavior.

## Why This Version Exists

v0.2.2 made missing model aliases explicit, but identity aliases are not enough when multiple Providers expose the same raw model ID. CLIProxyAPI's output model namespace is merged: a raw ID such as `gpt-5.5` cannot uniquely identify which Provider should be consumed.

v0.2.3 should make allmone-owned API-key Provider configuration produce both:

- A specific model ID: `<provider_id>-<raw_model_id>`.
- A shared raw model ID through `fork: true`, letting CLIProxyAPI keep its built-in behavior for raw model calls.

OAuth and auth-file Providers are intentionally excluded from allmone alias injection in this version. They should continue to use CLIProxyAPI's built-in alias and routing rules.

## Core Behavior

- Add a user-editable Provider id for API-key and OpenAI-compatible Provider entries.
- Restrict Provider id to ASCII letters, digits, and underscores: `^[A-Za-z0-9_]+$`.
- Require Provider id to be non-empty for entries whose model aliases are generated by allmone.
- Require Provider id uniqueness across all API-key and OpenAI-compatible Provider entries managed by allmone.
- Generate model aliases for API-key and OpenAI-compatible Provider entries as:

```text
<provider_id>-<raw_model_id>
```

- Keep the raw upstream model ID unchanged in `name`.
- Set `fork: true` for all allmone-generated API-key/OpenAI-compatible model alias rows.
- Preserve explicit user-edited alias rows only when they already match the v0.2.3 output contract or when the user intentionally edits the Provider id and the aliases are regenerated.
- Preserve existing Provider secrets, headers, base URL, disabled state, proxy URL, excluded models, and unknown CLIProxyAPI config fields.
- Do not write or sync OAuth/auth-file Provider aliases from allmone in this version.
- Keep OAuth/auth-file Provider model output delegated to CLIProxyAPI's built-in rules.
- Keep `Models` grouped by configured Provider and show both specific aliases and raw model IDs when `fork: true` makes both visible.

## Scope

### In Scope

- Provider id field for API-key upstream entries.
- Provider id field for OpenAI-compatible Provider entries.
- Main-process validation for Provider id format and uniqueness.
- Renderer validation and helpful error display for invalid Provider ids.
- Alias generation for API-key and OpenAI-compatible Provider models using `<provider_id>-<raw_model_id>`.
- Automatic `fork: true` on all allmone-generated API-key/OpenAI-compatible alias rows.
- Migration of existing identity aliases from v0.2.2 into Provider-specific aliases once a valid Provider id is present.
- Secret-safe persistence of Provider id through an allmone-owned sidecar in `~/.allmone/config.yaml`, overlaid onto Provider rows after CLIProxyAPI reads.
- Tests proving raw model calls remain visible through `fork: true` while specific Provider calls use generated aliases.

### Out Of Scope

- No allmone alias injection for OAuth/auth-file Providers.
- No auth-file scoped alias editor.
- No CLIProxyAPI routing, load-balancing, or priority implementation inside allmone.
- No provider protocol parsing or request/response transformation.
- No payload-rule UI.
- No broad v0.3.0 resource inventory fields.
- No automatic alias format for OAuth/auth-file Providers beyond what CLIProxyAPI already provides.

## Architecture

Add a Provider output identity boundary for allmone-managed API-key Provider entries.

Recommended changes:

- `src/main/upstreams/types.ts`: add a renderer-safe Provider id field to API-key/OpenAI-compatible Provider inputs and summaries.
- `src/main/upstreams/service.ts`: validate Provider id, preserve it on writes, and expose it in redacted summaries.
- `src/main/models/modelAlias.ts`: add helpers for generated aliases, Provider id validation, and `fork: true` projection.
- `src/main/models/service.ts`: update alias sync so API-key/OpenAI-compatible rows write generated aliases instead of identity aliases.
- `src/main/runtime/ipc.ts`: validate Provider id before calling services.
- `src/preload/index.ts` and `src/renderer/src/vite-env.d.ts`: expose the safe Provider id field.
- `src/renderer/src/pages/ProvidersPage.tsx`: add Provider id controls to API-key/OpenAI-compatible create/edit flows.
- `src/renderer/src/pages/ModelsPage.tsx`: keep display read-only, showing the final effective IDs produced by generated alias rows.

Storage is allmone-owned:

- Persist Provider id sidecar records in `~/.allmone/config.yaml`.
- Key sidecar records by Provider kind and entry index, because API-key values are secrets and must not become config keys.
- Continue writing `allmone.providerId` into CLIProxyAPI payloads when possible, but treat that metadata as opportunistic because current CLIProxyAPI writes may drop unknown fields.
- Overlay sidecar Provider ids onto CLIProxyAPI Provider entries before summaries, editing, and model alias sync.

## Alias Contract

For API-key and OpenAI-compatible Providers:

```yaml
- name: <raw_model_id>
  alias: <provider_id>-<raw_model_id>
  fork: true
```

Examples:

```yaml
- name: gpt-5.5
  alias: mimo_a-gpt-5.5
  fork: true

- name: moonshotai/kimi-k2:free
  alias: openrouter_work-moonshotai/kimi-k2:free
  fork: true
```

`provider_id` is normalized by validation only, not by silent rewriting. If the user enters invalid text, allmone should reject the save and ask for a valid id.

`raw_model_id` stays unchanged, including slashes, dots, colons, and hyphens returned by the upstream Provider.

For OAuth/auth-file Providers, allmone does not write this alias contract. CLIProxyAPI remains responsible for their output model names.

## Testing

Required verification:

```bash
bun run test
bun run typecheck
bun run build
```

Focused tests should cover:

- Provider id accepts only letters, digits, and underscores.
- Empty Provider id is rejected for API-key/OpenAI-compatible alias generation.
- Duplicate Provider ids are rejected across API-key and OpenAI-compatible Provider entries.
- API-key upstream sync writes `{ name, alias: "<provider_id>-<name>", fork: true }`.
- OpenAI-compatible sync writes `{ name, alias: "<provider_id>-<name>", fork: true }`.
- Existing v0.2.2 identity aliases migrate after a Provider id is available.
- Existing Provider secrets and unknown fields survive alias regeneration.
- OAuth/auth-file Providers do not receive allmone-generated aliases.
- `Models` display includes both generated alias and raw name when `fork: true`.
- Renderer payloads never expose provider API keys, bearer tokens, auth-file contents, or local output key plaintext.
